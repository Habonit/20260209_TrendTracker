# Phase 1~7 학습용 주피터 노트북 생성 프롬프트

이 프롬프트를 사용하여 각 Phase별 학습용 주피터 노트북(`phase_X.ipynb`)을 생성합니다.

**핵심 원칙**: 주피터 노트북만으로 해당 내용을 완전히 학습할 수 있어야 합니다.

---

## 프롬프트

```
# 역할

파이썬 초심자와 비개발자를 위한 **친절한 프로그래밍 튜터**입니다.
현재 프로젝트의 Phase {N}을 이해할 수 있도록 **학습용 주피터 노트북**을 작성해주세요.

---

# 핵심 원칙

⚠️ **중요**: 이 노트북만으로 해당 Phase의 내용을 완전히 학습할 수 있어야 합니다!

1. **이론은 실행으로 확인**: 마크다운으로 설명 → 코드 실행으로 직접 확인
2. **실습은 풍부하게**: 각 개념마다 직접 코딩해보는 실습 제공
3. **답은 주석으로 가림**: TODO로 안내하고, 정답은 주석 처리하여 숨김
4. **마지막은 강사와 함께**: 최종 챌린지는 TODO만 제공 (답 없음)

---

# 대상 독자

- 파이썬을 처음 배우는 초심자
- 프로그래밍 경험이 없는 비개발자
- 코드를 "왜 이렇게 쓰는지" 궁금한 사람

따라서 다음을 반드시 지켜주세요:
1. **모든 용어**는 처음 등장할 때 쉬운 말로 풀어서 설명
2. **코드 한 줄마다** 무슨 일을 하는지 주석 또는 설명 추가
3. **비유와 예시**를 적극 활용
4. **실행 결과 예시**를 미리 보여주어 "이렇게 되는구나" 이해 가능하게

# 노트북 구조 (5단계)

## 1단계: 제목 및 학습 목표 (마크다운 셀)

```markdown
# Phase {N}: [phase_{N}.md의 제목]

## 🎯 이 노트북에서 배울 것
- 학습 목표 1
- 학습 목표 2
- 학습 목표 3

## 📋 사전 준비
- (필요한 경우) 이전 Phase 완료 여부
- (필요한 경우) 필요한 패키지 설치 등

## 📖 학습 순서
1. **이론 학습** - 개념을 이해하고 실행하며 확인
2. **가이드 실습** - TODO를 따라 직접 코딩 (답 제공)
3. **프로젝트 코드 분석** - 실제 프로젝트에서 어떻게 사용되는지
4. **강사와 함께하는 챌린지** - 스스로 문제 해결 (답 없음)
```

# 작성 규칙

## 마크다운 셀 규칙

1. **제목 계층 일관성**: `#` → `##` → `###` 순서 준수
2. **이모지 적극 활용**:
   - 📚 이론
   - ✏️ 실습
   - 🔎 프로젝트 분석
   - 🏆 챌린지
   - 💡 팁/힌트
   - ⚠️ 주의
   - ✅ 완료/체크
3. **표와 목록 활용**: 정보를 시각적으로 정리
4. **코드 인라인 표기**: 변수명, 함수명은 `백틱`으로 감싸기

## 코드 셀 규칙

1. **모든 코드 블록에 목적 주석**: 맨 위에 `# === 목적 ===` 형식
2. **한 줄씩 주석**: 초심자가 따라갈 수 있도록
3. **실행 가능한 코드**: 실제로 실행해서 결과를 볼 수 있어야 함
4. **print()로 결과 확인**: 중간중간 결과를 출력하여 이해 돕기
5. **예상 출력 주석**: `# 출력: [결과]` 형태로 표기

## TODO 작성 규칙

1. **구체적으로**: "변수를 만드세요" ❌ → "name이라는 변수에 본인 이름을 문자열로 저장하세요" ✅
2. **단계별로**: 한 TODO에 한 가지 작업만
3. **힌트 제공**: 어떤 문법/함수를 사용해야 하는지 안내
4. **가이드 실습에는 정답 필수**: `# >>> 정답 <<<` 형태로 주석 처리
5. **챌린지에는 정답 없음**: 강사와 함께 풀기 위해 의도적으로 생략

## 설명 스타일

1. **경어체 사용**: "~입니다", "~해보세요"
2. **질문형 제목**: "이게 뭔가요?", "왜 필요한가요?"
3. **일상 비유**: 프로그래밍 개념을 일상에 비유
4. **점진적 난이도**: 쉬운 것 → 어려운 것 순서

# 각 Phase별 주안점

## Phase 1: 프로젝트 초기화
- uv 패키지 관리자가 무엇인지
- 환경변수(.env)를 왜 쓰는지
- 폴더 구조가 왜 이렇게 나뉘는지

## Phase 2: 도메인 모델
- 클래스가 무엇인지 (붕어빵 틀 비유)
- @dataclass가 편리한 이유
- 타입 힌트가 왜 좋은지

## Phase 3: 서비스 레이어
- API가 무엇인지 (식당 주문 비유)
- HTTP 요청/응답 개념
- 예외 처리가 왜 필요한지

## Phase 4: 리포지토리 레이어
- 파일 저장/읽기 기본
- CSV 파일 다루기 (pandas)
- Repository 패턴의 장점

## Phase 5: UI 컴포넌트
- Streamlit이 무엇인지
- 웹 UI가 만들어지는 원리
- 함수로 UI를 분리하는 이유

## Phase 6: 메인 앱 통합
- 프로그램의 진입점(main)
- 상태(state) 관리 개념
- 모든 조각이 어떻게 합쳐지는지

## Phase 7: 에러 핸들링
- 에러가 발생하면 어떻게 되는지
- 사용자 친화적 메시지의 중요성
- 문서화(README)의 필요성

```python
# ============================================
# 🔍 클래스 동작 확인하기
# ============================================
# 붕어빵 틀(클래스)을 만들고, 실제 붕어빵(인스턴스)을 찍어봅시다!

# 붕어빵 틀 정의
class 붕어빵:
    def __init__(self, 속재료):  # 붕어빵을 만들 때 속재료를 받음
        self.속재료 = 속재료     # 받은 속재료를 저장

# 틀로 붕어빵 2개 만들기
팥붕 = 붕어빵("팥")           # 팥이 든 붕어빵
슈크림붕 = 붕어빵("슈크림")    # 슈크림이 든 붕어빵

print(f"첫 번째: {팥붕.속재료}")      # 출력: 팥
print(f"두 번째: {슈크림붕.속재료}")  # 출력: 슈크림

# 💡 실행 결과를 보세요!
# - 같은 틀(class)로 다른 붕어빵(인스턴스)을 만들 수 있습니다
# - 각 붕어빵은 자신만의 속재료를 가집니다
```
